Небольшой **плагин для 1C:EDT**, который отдаёт **живую семантическую модель** EDT
AI-агентам и другим инструментам по протоколу **MCP**.

Статические парсеры читают тексты; EDT-Bridge вместо этого спрашивает работающую IDE. Он
отвечает на то, что требует *живой* модели: собственные диагностики EDT, реальную структуру и
типы метаданных, семантические перекрёстные ссылки, **валидацию запроса против настоящих
метаданных проекта** и **Синтакс-помощник платформы** из поставки EDT – плюс инструменты записи,
которые создают, рефакторят, собирают и поставляют, всё через штатный движок EDT.

> Чтение **и** запись. Только localhost; запись под токеном и по умолчанию dry-run. Плагин
> работает внутри EDT – EDT (GUI или headless) должна быть запущена с вашим проектом.


Заметки о разработке и новости – в Telegram-канале [1С × ИИ: инженерный цех](https://t.me/ceh_1c_ai).

## Установка (рекомендуется: pipx)

Одна команда ставит всё – и обвязку клиента, и плагин. Обвязка
[**edt-bridge-mcp**](../../python/README.ru.md) – это stdio MCP-сервер, с которым разговаривает клиент;
она пробрасывает запросы в запущенную EDT, **сама поднимает headless EDT**, когда ни одна не
открыта, и **сама доставляет jar плагина** в `dropins/` EDT, если его нет. Никакой jar руками
копировать не нужно.

```bash
pipx install edt-bridge-mcp
```

Затем зарегистрируйте её в MCP-клиенте (показан Claude Code; `--workspace` – workspace EDT,
который поднимать при автозапуске headless):

```bash
claude mcp add edt-bridge -- edt-bridge-mcp --workspace "D:\\путь\\к\\workspace-EDT"
```

Это вся настройка. Флаги обвязки, токен write-инструментов и `self-update` описаны в
[python/README.ru.md](../../python/README.ru.md). Хотите запускать плагин сами, без обвязки? См.
[Установку вручную](#установка-вручную-без-обвязки) ниже.

<details>
<summary>Нет <code>pipx</code>?</summary>

`pipx` ставит Python-приложения CLI в изолированные окружения. Установить один раз:

```bash
python -m pip install --user pipx
python -m pipx ensurepath      # затем переоткрыть терминал
```

macOS: `brew install pipx && pipx ensurepath`. Подробнее: <https://pipx.pypa.io>.
</details>

### Настройки в самой EDT

У плагина есть своя страница параметров – **Окно ▸ Параметры ▸ EDT-Bridge**; именно оттуда берётся
токен, когда мост работает внутри GUI EDT:

| Параметр | Что это |
|----------|---------|
| **Токен для инструментов записи** | Общий секрет, который требует каждый инструмент записи. Пустой означает, что запись запрещена, а не что она без охраны. То же значение уходит клиенту как `EDT_BRIDGE_TOKEN`. |
| **Порт MCP-сервера** | По умолчанию 8770. **Вступает в силу после перезапуска EDT** – работающий сервер до этого держит прежний порт. |
| **Разрешить выполнение произвольного кода BSL при отладке** | По умолчанию выключено. Управляет `edt_evaluate`, который исполняет код против живой информационной базы, – только тестовые стенды. |

**Параметр запуска сильнее этой страницы.** Системные свойства `-Dedt.bridge.*` и переменные окружения
`EDT_BRIDGE_*`, заданные при старте, имеют приоритет над сохранёнными значениями – так обвязка
управляет headless-экземпляром, и по этой же причине заданный здесь токен может выглядеть
проигнорированным, если он был передан ещё и в командной строке.

## Рядом

EDT-Bridge работает с 1С:Предприятием. Соседняя платформа, 1С:Элемент, обслуживается парой
инструментов на том же принципе – инструмент отдаёт агенту руки, а не советы:

- **[XBSL](https://docs.keyfire.ru/xbsl/ru/)** – линтер исходников Элемента с автоправками, LSP-сервер,
  скаффолдинг метаданных, MCP-сервер и расширение VS Code.
- **[Elemctl](https://docs.keyfire.ru/elemctl/ru/)** – доставка на стенд Элемента: сборка, загрузка,
  применение и проверка, что оно применилось.

## Дашборд

Откройте `http://127.0.0.1:8770/` в браузере – встроенный дашборд: статус сервера, открытые
проекты EDT и интерактивный запуск любого инструмента. Светлая/тёмная тема и переключатель EN/RU.
