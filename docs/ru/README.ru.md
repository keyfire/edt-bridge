# edt-bridge

Небольшой **плагин для 1C:EDT**, который отдаёт **живую семантическую модель** EDT
AI-агентам и другим инструментам по протоколу **MCP**.

Статические парсеры читают тексты; edt-bridge вместо этого спрашивает работающую IDE. Он
отвечает на то, что требует *живой* модели: собственные диагностики EDT, реальную структуру и
типы метаданных, семантические перекрёстные ссылки, **валидацию запроса против настоящих
метаданных проекта** и **Синтакс-помощник платформы** из поставки EDT – плюс инструменты записи,
которые создают, рефакторят, собирают и поставляют, всё через штатный движок EDT.

> Чтение **и** запись. Только localhost; запись под токеном и по умолчанию dry-run. Плагин
> работает внутри EDT – EDT (GUI или headless) должна быть запущена с вашим проектом.

## Установка (рекомендуется: pipx)

Одна команда ставит всё – и обвязку клиента, и плагин. Обвязка
[**edt-bridge-mcp**](../../python/README.ru.md) – это stdio MCP-сервер, с которым разговаривает клиент;
она пробрасывает запросы в запущенную EDT, **сама поднимает headless EDT**, когда ни одна не
открыта, и **сама доставляет jar плагина** в `dropins/` EDT, если его нет. Никакой jar руками
копировать не нужно.

```bash
pipx install edt-bridge-mcp
```

Затем зарегистрируйте её в MCP-клиенте (показан Claude Code; `--workspace` – workspace EDT,
который поднимать при автозапуске headless):

```bash
claude mcp add edt-bridge -- edt-bridge-mcp --workspace "D:\\путь\\к\\workspace-EDT"
```

Это вся настройка. Флаги обвязки, токен write-инструментов и `self-update` описаны в
[python/README.ru.md](../../python/README.ru.md). Хотите запускать плагин сами, без обвязки? См.
[Установку вручную](#установка-вручную-без-обвязки) ниже.

<details>
<summary>Нет <code>pipx</code>?</summary>

`pipx` ставит Python-приложения CLI в изолированные окружения. Установить один раз:

```bash
python -m pip install --user pipx
python -m pipx ensurepath      # затем переоткрыть терминал
```

macOS: `brew install pipx && pipx ensurepath`. Подробнее: <https://pipx.pypa.io>.

## Инструменты

Инструменты – `edt_*` (snake_case); параметры – camelCase (`projectName`, `fqn`, `queryText`);
поддерживаются кириллические FQN (`Справочник.Контрагенты`). Префикс `edt_` сознательный:
MCP-хост показывает агенту инструменты всех серверов одним списком, и имя должно нести контекст
само – `edt_rename` однозначен там, где голый `rename` опасно многозначен.

### Чтение

| Инструмент | Что возвращает |
|------------|----------------|
| `edt_projects` · `edt_project_errors` | Открытые проекты рабочей области – имя, путь на диске, natures, является ли проектом 1C:EDT – и проблемы валидации EDT по проекту (сообщение, severity, ресурс, строка). С этого стоит начинать: они показывают, к чему вообще можно обращаться. |
| `edt_metadata_objects` · `edt_metadata_details` | Объекты метаданных верхнего уровня с опциональным фильтром по типу и подстроке имени; затем свойства одного объекта **и его структура** – реквизиты, табличные части, формы, команды, макеты, измерения, ресурсы, значения перечислений – с типом каждого реквизита. |
| `edt_find_references` · `edt_outgoing_calls` · `edt_outgoing_structures` | Куда идут вызовы. Входящие ссылки на объект метаданных из индекса EDT (с `method` – BSL-места вызова `CommonModule.X.method`); обратное направление, какие методы ВЫЗЫВАЕТ модуль / метод / форма на уровень наружу, с числом мест вызова и флагом слоя ExtAPI; и, best-effort, ключи верхнего уровня `Структуры`, переданной каждому квалифицированному исходящему вызову. |
| `edt_module_text` · `edt_go_to_definition` · `edt_symbol_info` | Чтение BSL: исходный текст модуля (или одного метода) со списком процедур и функций и их сигнатурами, по FQN или modulePath; определение символа в позиции (вид цели, имя, объект-владелец, расположение); тип символа в позиции – элемент под курсором и вычисленные типы значения выражения. |
| `edt_search_modules` | Полнотекстовый поиск по модулям BSL проекта – подстрока или регулярное выражение, опциональный фильтр пути. Чтение идёт через буферы файлов Eclipse, поэтому модуль, открытый в редакторе, ищется в текущем виде, вместе с несохранёнными правками. Если `edt_find_references` отвечает "кто вызывает этот метод", то этот – "где встречается такой текст". |
| `edt_validate_query` | Валидирует запрос 1С против живых метаданных: синтаксис **и** семантику (несуществующие таблицы/поля, ошибки типов) с позициями. |
| `edt_form_structure` · `edt_form_render` · `edt_picture_export` | Формы и изображения: дерево элементов управляемой формы с привязками данных, статическими visible/enabled/readOnly, обработчиками, свойствами полей ввода, связкой команда→кнопка и условным оформлением, плюс реквизиты, команды, параметры и обработчики; та же форма, отрисованная в PNG штатным offscreen-рендером EDT (вариант интерфейса и тема на выбор); содержимое CommonPicture из Picture.zip. |
| `edt_infobases` · `edt_platform_installations` | С чем работает платформенная сторона: зарегистрированные в EDT информационные базы (имя, uuid, строка соединения) с привязками открытых проектов, и установки 1С:Предприятие, из которых EDT выбирает при выгрузке `.epf`/`.erf` и создании базы – каждая разрешена до конкретной установки с толстым клиентом, плюс полные установки, найденные на диске. |
| `edt_platform_help` | **Синтакс-помощник платформы** 1С:Предприятие из поставки EDT (реальная справка API – объекты, методы, свойства, события, рус+англ): поиск по имени или чтение страницы текстом. Свериться с фактическим API вместо угадывания сигнатур. |

### Запись

Инструменты записи меняют модель через штатный движок EDT (не текстовой заменой). Все –
**под токеном** и по умолчанию **dry-run** (`apply=false` возвращает план и ничего не меняет);
`apply=true` применяет изменение и сериализует `.mdo`.

| Инструмент записи | Что делает |
|-------------------|------------|
| `edt_create_object` · `edt_delete_object` | Создаёт новый топ-объект (Справочник/Документ/Перечисление/РегистрСведений/...) через фабрику EDT и инициализатор типа, с регистрацией в Configuration – либо удаляет объект с **каскадным удалением всех ссылок в метаданных И в BSL** (нужен `force`). |
| `edt_add_attribute` · `edt_modify_attribute` · `edt_remove_attribute` | Добавляют реквизит объекту метаданных (тип / синоним / comment, с валидацией), меняют тип, синоним или comment существующего, либо удаляют его – удаление проверяет ссылки и отказывает, пока они есть, без `force`. |
| `edt_rename` | Переименовывает объект или член с **каскадом всех ссылок в метаданных И в BSL** через штатный движок рефакторинга (нужен `force` – переименование ломает совместимость). |
| `edt_add_method` · `edt_delete_method` | Добавляют и удаляют процедуру/функцию в BSL модуля, обе операции по модели. Вставка отказывает, если результат не парсится; вырезание забирает прилегающий документирующий комментарий, а dry-run возвращает точный удаляемый текст (нужен `force` – удаление кода деструктивно). |
| `edt_add_form` | Добавляет управляемую форму объекту метаданных штатным генератором EDT – тем же, что стоит за мастером "Новая форма", – поэтому форма, её элементы и модуль генерируются, а не пишутся руками в XML. |
| `edt_add_form_attribute` · `edt_modify_form_attribute` · `edt_remove_form_attribute` | Добавляют, меняют и удаляют реквизит формы – либо, с `columnOf`, колонку реквизита типа ТаблицаЗначений. Идентификаторы выдаёт служба идентификаторов формы; помимо грамматики типов метаданных принимаются платформенные типы, допустимые у формы (`ТаблицаЗначений`, `СписокЗначений`, ...). Удаление перечисляет связанные элементы и требует `force`. |
| `edt_add_form_command` · `edt_modify_form_command` · `edt_remove_form_command` | Добавляют, меняют и удаляют команду формы. При добавлении можно сразу записать заготовку процедуры-обработчика в модуль формы, создав его, если модуля не было. Удаление перечисляет связанные кнопки и требует `force`. |
| `edt_add_form_item` · `edt_modify_form_item` · `edt_remove_form_item` | Добавляют, меняют и удаляют элементы формы – поле, таблицу, кнопку, группу, декорацию – через штатный `IFormItemManagementService`, тот самый, который вызывает редактор форм. Таблица, связанная с реквизитом типа ТаблицаЗначений, получает колонки автоматически. Удаление забирает всё вложенное и требует `force`. |
| `edt_adopt_object` | Заимствует объект базовой конфигурации в проект расширения штатным `IModelObjectAdopter` EDT – шаг, без которого расширение не может ничего перехватить у этого объекта, и который достраивает `edt_create_extension`. |
| `edt_create_extension` · `edt_create_external_object` | Начало работы над проектом. Проект **расширения конфигурации** к базовому через `IExtensionProjectManager`, с префиксом имён и назначением (Адаптация·Дополнение·Исправление) на свежем детаченном Configuration расширения – либо проект **внешней обработки**, начало цикла "обработка → .epf". |
| `edt_clean_project` · `edt_delete_project` | Завершение работы над проектом. Сбросить результаты сборки, чтобы валидация отработала заново (диалог "Очистить" из EDT, программно: возвращает число замечаний до и после, дождавшись, пока оно перестанет меняться, – устаревший маркер, переживший свою причину, хуже отсутствия маркера), либо удалить проект из workspace средствами Eclipse, чтобы не осталось фантома (нужен `force` – удаление необратимо). |
| `edt_build_extension` · `edt_dump_external_object` | Сборка бинарников: **`.cfe`** из проекта расширения и **`.epf`/`.erf`** из внешней обработки или отчёта. Обе умеют обойти резолвер платформы EDT, когда тот не даёт толстый клиент: модель выгружается в XML конструктора в процессе, затем артефакт собирается полной установкой 1С с диска во временной базе, которая потом удаляется. `logPath` кладёт лог сборки рядом с артефактом. |
| `edt_create_infobase` · `edt_update_infobase` · `edt_register_platform` | Информационные базы и платформа за ними: создать **пустую файловую базу** и зарегистрировать её в списке EDT (с фоллбеком на полную установку с диска, если под версию у EDT ничего нет); обновить конфигурацию базы **из проекта EDT** штатным механизмом синхронизации (изменения структуры БД подтверждаются автоматически, конфликт прерывает); зарегистрировать полную установку в EDT, чтобы ею мог пользоваться её собственный движок. |

### Отладка

Подключение к debug-серверу (dbgs) **запущенной** ИБ и управление выполнением. Использовать
тестовый стенд, не продакшен. Все под токеном; `edt_evaluate` – жёстче всех.

| Инструмент отладки | Что делает |
|--------------------|------------|
| `edt_debug_attach` · `edt_debug_detach` | Подключают сессию отладки к debug-серверу запущенной ИБ (возвращая `sessionId` для остальных debug-инструментов) и отключают её, завершая сессию и освобождая базу. |
| `edt_debug_inspect` · `edt_debug_control` | Список потоков сессии и, для **приостановленных**, кадры стека BSL с переменными верхнего кадра (только чтение); затем управление выполнением – `suspend`/`resume` либо `stepOver`/`stepInto`/`stepReturn` приостановленного потока. |
| `edt_evaluate` | Вычисляет **произвольное BSL-выражение** в приостановленном кадре – исполнение кода против живой ИБ. Нужен токен **и** per-call `allowCodeExecution=true` **и** серверный переключатель `EDT_BRIDGE_ALLOW_EVALUATE=1` (по умолчанию выключено). |

## Требования

**Чтобы пользоваться мостом**

- **1C:EDT** с открытым проектом – либо позвольте `edt-bridge-mcp` поднять headless EDT.
- Для `edt_dump_external_object` (сборка `.epf`/`.erf`) и `edt_update_infobase`: установленная
  локально **платформа 1С:Предприятие**, совместимая с версией проекта – EDT управляет ею для
  компиляции бинарника и обновления информационной базы.

**Чтобы собрать плагин из исходников** (для контрибьюторов – конечные пользователи ставят через pipx)

- **JDK 17+** (байткод бандла нацелен на Java 17, рантайм EDT).
- Локальный **пул бандлов EDT**. На **Windows** p2-пул `%USERPROFILE%\.p2\pool\plugins`; на
  **macOS** пул внутри установленного компонента
  `.../1C/1CE/components/1c-edt-<ver>-x86_64/1cedt (<ver>).app/Contents/Eclipse/plugins`
  (shell-сборка определяет его сама).

## Установка вручную (без обвязки)

Обвязка pipx доставляет jar и запускает EDT за вас. Чтобы запускать плагин самому:

1. Возьмите jar – со [страницы Releases](https://github.com/keyfire/edt-bridge/releases) (с
   `SHA256SUMS.txt`) или соберите (ниже).
2. Скопируйте в `dropins/` EDT – Windows `.../installations/<EDT>/1cedt/dropins/`, **macOS**
   `.../1c-edt-<ver>-x86_64/1cedt (<ver>).app/Contents/Eclipse/dropins/` (создайте, если нет).
   Держите там только один jar edt-bridge – два заставят Equinox взять произвольный.
3. **Перезапустите EDT.** Плагин поднимает MCP-сервер на `http://127.0.0.1:8770/mcp` (или на
   следующем свободном порту, если 8770 занят).

Запустить EDT **headless** (без GUI): `scripts/run-headless.ps1 -Workspace <ws>` (Windows) или
`scripts/run-headless.sh --workspace <ws>` (macOS / Linux); `scripts/toggle-headless.ps1` включает/выключает
одним действием. Запущенную GUI EDT это никогда не трогает. Запустить **GUI** на рабочей области:
`scripts/run-gui.ps1 -Workspace <ws>`.

Оба лаунчера отказываются поднимать вторую EDT на уже занятой рабочей области и не снимают
блокировку, которую держит живой экземпляр, – общая проверка лежит в `scripts/edt-common.ps1`.
Запуск `1cedt.exe` руками эту проверку минует: второй экземпляр падает с "рабочая область уже
используется", а если сделать так дважды – остаётся ворох недозапущенных окон.

MCP-клиент может говорить с плагином и напрямую по HTTP (без обвязки) – добавьте
`{ "edt-bridge": { "type": "http", "url": "http://127.0.0.1:8770/mcp" } }` в его `.mcp.json`.
Сервер говорит простым JSON-RPC по HTTP (`initialize` / `tools/list` / `tools/call`).

### Сборка из исходников

Без Maven (быстрее всего – локальный JDK + пул EDT, без сети):

```powershell
# Windows – по умолчанию: -Pool %USERPROFILE%\.p2\pool\plugins, -JdkHome %JAVA_HOME%
powershell -ExecutionPolicy Bypass -File scripts/build-nomaven.ps1
```

```bash
# macOS / Linux – --pool автоопределяется из пула установленного 1C:EDT
./scripts/build-nomaven.sh
```

Даёт `build/io.github.keyfire.edtbridge_<версия>.<таймстамп>.jar`. Maven + Tycho
(`mvn -f pom.xml clean verify`, сначала правьте `edt-bridge.target`) доступен для CI.

Релизы собираются из локально построенного jar – CI не может его скомпилировать (SDK-бандлы
1C:EDT проприетарны и не скачиваются анонимно). Мейнтейнер запускает `scripts/build-nomaven.ps1 -Dist`,
кладёт jar в `dist/`, ставит тег `vX.Y.Z` и пушит тег; `.github/workflows/release.yml`
прикладывает jar + контрольную сумму. Проверить ассет можно пересборкой из тегированного
исходника и сравнением.

Откройте `http://127.0.0.1:8770/` в браузере – встроенный дашборд: статус сервера, открытые

## Ошибки и обратная связь

edt-bridge – молодой инструмент, и его возможности могут содержать ошибки и шероховатости. Если
что-то ведёт себя неожиданно, падает или у вас есть пожелание – заведите
**[issue на GitHub](https://github.com/keyfire/edt-bridge/issues)**: опишите, что делали, что
ожидали и что ответил инструмент (или приложите соответствующую строку лога EDT).

## Безопасность

- Слушает **только `127.0.0.1`** – никогда публичный интерфейс.
- **Запись под воротами**: каждый write-инструмент требует настроенный токен, по умолчанию
  dry-run и работает только с локальной моделью EDT; `edt_rename`, `edt_delete_object` и
  `edt_delete_method` дополнительно требуют явный `force`.
- Опциональный **токен-секрет** – задайте `EDT_BRIDGE_TOKEN` (или `-Dedt.bridge.token=`) и шлите
  `Authorization: Bearer <token>` (или `X-Edt-Bridge-Token: <token>`). Порт доступен любому
  локальному процессу – на общих машинах задавайте токен.
- Порт: `EDT_BRIDGE_PORT` / `-Dedt.bridge.port=` (по умолчанию 8770; при занятости берётся
  следующий свободный).
- Нашли проблему с безопасностью? Сообщите **приватно** – см. [SECURITY.md](SECURITY.ru.md), а не
  публичным issue.
