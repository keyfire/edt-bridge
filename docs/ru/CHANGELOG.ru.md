# История изменений

Все заметные изменения edt-bridge фиксируются здесь.

Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/), проект следует
[семантическому версионированию](https://semver.org/lang/ru/). Jar плагина и обвязка
`edt-bridge-mcp` используют один номер версии.

## [0.4.0] – 2026-07-19

### Добавлено
- `edt_clean_project` – сбросить результаты сборки проекта, чтобы валидация отработала заново;
  программный аналог диалога "Очистить" в EDT. Проверки EDT висят на сборке, поэтому именно это
  заставляет их переиграть: иначе замечание переживает исправление кода, а устаревший маркер хуже,
  чем никакого. Возвращает число замечаний до и после, дождавшись, пока оно перестанет меняться.
- `edt_add_form` – создание управляемой формы у существующего объекта метаданных. Инструмент
  использует штатный `IFormGenerator` EDT – движок мастера "Новая форма" – и повторяет
  последовательность `FormNewWizardRelatedModelsFactory`: создать запись `BasicForm`, связать её со
  свойством `forms` владельца, сгенерировать содержимое формы, связать пару через `setMdForm`,
  подключить форму как топ-объект BM под FQN от `ITopObjectFqnGenerator`, затем записать модуль из
  `generateModuleContent`. Вид формы принимается и как имя `FormType`, и как русский синоним
  (`ФормаЭлемента`, `ФормаСписка`, `ФормаЗаписи`, ...); если вид не задан, берётся обычный для
  владельца. По умолчанию dry-run, под токеном, аддитивно – force не нужен.
- Элементы формы: `edt_add_form_item`, `edt_modify_form_item`, `edt_remove_form_item`. Элементы
  создаются штатным `IFormItemManagementService` – тем самым, который вызывает редактор форм, –
  поэтому имена, идентификаторы, фактический вид поля и автозаполнение колонок таблицы определяет
  EDT, а не ручная сборка. Виды: поле, таблица, кнопка, группа (обычная группа, страница, страницы,
  командная панель, группа кнопок, группа колонок, подменю) и декорация. Dry-run проверяет путь к
  данным по реквизитам формы, а команду – по её командам, поэтому неверная связка роняет план.
  Удаление забирает всё вложенное и требует `force`.
- Члены формы, полный набор для обоих видов: `edt_add_form_attribute`, `edt_modify_form_attribute`,
  `edt_remove_form_attribute`, `edt_add_form_command`, `edt_modify_form_command`,
  `edt_remove_form_command`. Идентификаторы выдаёт штатный `FormIdentifierService`. Типы реквизитов
  используют грамматику `edt_add_attribute` и дополнительно принимают платформенные типы, допустимые
  у формы (`ТаблицаЗначений`, `СписокЗначений`, ...), которые резолвятся через провайдер
  платформенных типов; тип теперь строится ещё в dry-run, поэтому негодная строка типа роняет план,
  а не применение. `edt_add_form_command` умеет записать заготовку процедуры-обработчика в модуль
  формы, создав его, если модуля не было. Удаления перечисляют, что ещё связано с членом, – элементы
  для реквизита, кнопки для команды – и требуют `force`. Три инструмента реквизита принимают также
  `columnOf`: он адресует КОЛОНКУ реквизита типа ТаблицаЗначений вместо реквизита формы – колонка
  это тот же `AbstractFormAttribute`, поэтому отдельных инструментов ей не нужно. Когда колонки
  заданы, поля колонок таблицы ставятся через `edt_add_form_item` (`parent` – таблица, `dataPath` –
  `реквизит.колонка`).
- `edt_delete_project` – удаление проекта из workspace, замыкающее цикл, который открывают
  инструменты создания проектов. Удаление идёт через workspace Eclipse, поэтому дерево ресурсов
  обновляется; если снести каталог проекта руками, остаётся фантом – проект возвращается из снимка
  дерева при следующем старте и держит имя занятым. `deleteContent` выбирает между снятием
  регистрации и стиранием файлов. По умолчанию dry-run, для применения нужен `force`.
- `edt_create_external_object` принимает `scriptVariant` (Russian / English) и применяет его через
  `IExternalObjectProjectManager.setScriptVariant` / `setLanguages`. EDT создаёт standalone-проект
  с английским вариантом, из-за чего генерируемые члены получались `Object`, а не `Объект`; dry-run
  теперь предупреждает, если вариант не задан у standalone-проекта.

### Исправлено
- `edt_dump_external_object` снова собирает `.epf`/`.erf` там, где EDT не может. EDT резолвит
  НОВЕЙШУЮ сборку линии версии, поэтому линия, во главе которой стоят тонкие клиенты, роняла штатный
  dumper с `MatchingRuntimeNotFound` даже при наличии полных установок этой линии. Добавлен дисковый
  маршрут - экспорт в XML конфигуратора внутри процесса, затем сборку выполняет полная платформа,
  найденная на диске, - по образцу дискового фоллбека создания информационной базы. Заодно перестал
  врать dry-run: штатная `validateDumpGeneration` возвращает OK независимо от наличия толстого
  клиента, поэтому платформа проверяется напрямую, а план называет маршрут. Новый `logPath` пишет лог
  сборки рядом с артефактом (по умолчанию `<targetPath>.log`) - именно там платформа объясняет
  причину отказа.
- Модули внешних объектов резолвятся по FQN. В карте "тип метаданных - каталог" не было внешних
  объектов, поэтому `ExternalDataProcessor.X.Form.Y` отвергался с "unsupported owner type for a form"
  в `edt_add_method`, `edt_delete_method` и `edt_module_text` (обходилось передачей `modulePath`).
  Теперь `ExternalDataProcessor` и `ExternalReport` в карте есть. Расширений конфигурации это никогда
  не касалось - раскладка их исходников совпадает с конфигурацией.

### Изменено
- MCP-сервер сообщает собственную версию плагина, взятую из манифеста бандла, вместо зашитой строки:
  релиз поднимает её в одном месте, и страница состояния с рукопожатием не могут разъехаться.

## [0.3.1] – 2026-07-19

### Исправлено
- Обвязка stdio фиксирует стандартные потоки в UTF-8. На Windows они открывались в кодировке ANSI,
  и символ "→" в описании `edt_modify_attribute` ронял `tools/list` ошибкой кодирования `charmap` –
  клиент в результате не получал ни одного инструмента. Кириллица в аргументах на входе была
  подвержена тому же дефекту.
- В таблицах инструментов README не хватало четырёх инструментов, добавленных в 0.3.0
  (`edt_platform_installations`, `edt_register_platform`, `edt_create_infobase`,
  `edt_build_extension`) – теперь они перечислены.

## [0.3.0] – 2026-07-19

### Добавлено
- `edt_platform_installations` – показывает установки платформы 1С:Предприятие, которые EDT
  резолвит (для сборки бинарников и создания баз), включая полные установки, найденные на диске.
- `edt_create_infobase` – создание пустой файловой базы через EDT, с фоллбеком на поиск подходящей
  полной установки платформы на диске, если у EDT её нет.
- `edt_register_platform` – регистрация установки платформы в EDT.
- `edt_build_extension` – сборка расширения конфигурации (`.cfe`) через `ibcmd`: экспорт проекта
  средствами EDT и упаковка во временной базе, которая затем удаляется.
- Страница настроек EDT-Bridge (токен, порт MCP-сервера и переключатель evaluate), чтобы GUI-EDT,
  запущенная из обычного ярлыка, могла аутентифицировать инструменты записи.
- Раздел "Ошибки и обратная связь" в README; workflow trusted publishing для PyPI.

### Исправлено
- `edt_infobases` теперь разворачивает группы баз – групповые/серверные базы больше не теряются.

### Изменено
- Внутреннее: единый модельный шлюз разбит на сфокусированные гейтвеи по областям (проекты, чтение
  и запись метаданных, формы, платформа, отладка, BSL, документация) плюс общие хелперы. Чистый
  рефакторинг – поведение не меняется.

## [0.2.1] – 2026-07-18

### Изменено
- Сплошная типографика по стилю проекта (среднее тире вместо длинного, три точки вместо символа
  многоточия), включая описания инструментов и Python-обвязку.

## [0.2.0] – 2026-07-18

### Добавлено
- `edt_platform_help` – Синтакс-помощник платформы 1С:Предприятие из поставки EDT (реальная справка
  API, с поиском, рус + англ).
- Многопоточный MCP-сервер (ограниченный пул воркеров).
- Фоллбек порта: если 8770 занят, берётся следующий свободный, и обвязка его находит.

### Исправлено
- `apply` создания проектов теперь работает надёжно (строится и привязывается свежий детаченный
  Configuration / внешняя обработка, а не роняется жизненный цикл headless-EDT).

## [0.1.1] – 2026-07-18

### Добавлено
- Инструменты записи цикла создать / разработать / собрать / доставить (по умолчанию dry-run).

### Исправлено
- `apply` создания проектов приведён к жизненному циклу headless-EDT.

## [0.1.0] – 2026-07-18

### Добавлено
- Полный цикл создать / разработать / собрать / доставить, открытый по MCP.
- Stdio-обвязка `edt-bridge-mcp` (pipx): пробрасывает запросы в запущенную EDT, сама поднимает
  headless-EDT, когда ни одна не открыта, и доставляет jar плагина в `dropins/` EDT.

## [0.0.1] – 2026-07-11

### Добавлено
- Первый выпуск: инструменты чтения по MCP (проекты, ошибки валидации, свойства и перечень
  метаданных, ссылки, валидация запросов, формы), встроенный дашборд и localhost MCP-сервер.
